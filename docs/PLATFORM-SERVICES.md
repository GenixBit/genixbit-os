# GenixBit OS Platform Services

## Official Service Map

| Service | Intended Domain | Purpose | Current Repository Status |
| --- | --- | --- | --- |
| Product website | `https://os.genixbit.com` | product information, editions, downloads, system requirements, release status | deployment pending |
| Documentation | `https://docs.os.genixbit.com` | installation, development, AI, server, creator, troubleshooting, release, and security documentation | deployment pending |
| Package repository | `https://packages.os.genixbit.com` | signed GenixBit APT packages and repository metadata | infrastructure pending; do not advertise as active |
| Source code | `https://github.com/GenixBit/genixbit-os` | GPL-covered OS builder source, governance, roadmap, and validation records | active |
| Company | `https://www.genixbit.com` | GenixBit Labs Private Limited corporate website | external company service |

Additional services such as downloads, update metadata, app catalog, status, and publisher tools should receive separate subdomains only after their ownership and security model are approved.

## Recommended Service Separation

### Public Website

- static-first deployment;
- no database required for the initial release;
- original GenixBit content and visual design;
- release status sourced from a reviewed manifest;
- no fake download button before a validated ISO exists;
- privacy-respecting analytics only if explicitly approved.

### Documentation Service

- versioned documentation;
- stable URLs for installation and troubleshooting;
- separate current, alpha, and archived release documentation;
- build from Markdown stored in a GenixBit-controlled repository;
- search index generated during deployment;
- clear distinction between available and planned features.

### Package Repository

- isolated from the public website process;
- repository metadata generated by a Debian repository tool such as aptly or an equivalent reviewed solution;
- offline or tightly controlled signing key workflow;
- only public verification keys stored on public servers;
- snapshots, promotion channels, rollback, backup, and audit logs;
- `alpha`, `testing`, and `stable` channels;
- read-only public HTTP serving layer;
- no private signing key mounted into the public web container.

### Downloads

- ISO images, torrent files, checksums, signatures, and release manifests;
- immutable release paths;
- CDN/object storage after initial validation;
- release manifest containing file name, size, SHA-256, architecture, channel, build commit, and publication date;
- no public release until baseline boot and installation tests pass.

### App and Model Catalog

- signed catalog metadata;
- no direct root command strings accepted from an untrusted remote catalog;
- installation adapters implemented and reviewed in the client;
- license, publisher, source, permissions, architecture, and support state displayed;
- separate review process for applications and AI models.

## Environment Layout

### Development

Local preview, fake-free status data, no production credentials, and no official package signing.

### Staging

Private or access-controlled environment using staging DNS, test certificates, test packages, non-production keys, and preview content.

### Production

GenixBit-controlled server, DNS, TLS, backups, monitoring, restricted administrative access, firewall, patching, and documented recovery procedures.

## Initial Single-Server Layout

A single new Linux server can host the initial static product website, static documentation preview, and package-service placeholder using containers and Caddy. The actual signed APT repository should remain disabled until package signing and publication are implemented safely.

Recommended minimum for the initial web preview:

- Ubuntu LTS server on amd64;
- 2 vCPU;
- 4 GB RAM;
- 40 GB SSD;
- public IPv4 address;
- ports 22, 80, and 443 controlled by firewall;
- SSH key authentication;
- non-root deployment user;
- Docker Engine and Compose plugin;
- DNS A/AAAA records controlled by GenixBit;
- automated backups for configuration and content.

This is separate from the Ubuntu 26.04 amd64 machine required to build the OS ISO.

## DNS Plan

Point the following records to the new web server only after it is provisioned and hardened:

```text
os.genixbit.com             A      <NEW_SERVER_IPV4>
docs.os.genixbit.com        A      <NEW_SERVER_IPV4>
packages.os.genixbit.com    A      <NEW_SERVER_IPV4>
```

Add AAAA records only when IPv6 is configured and firewalled correctly.

## TLS and Reverse Proxy

The repository includes an initial Caddy-based preview stack under [`deploy/`](../deploy/). Automatic public TLS will work only after DNS resolves to the server and ports 80/443 are reachable.

## Required Production Controls

- operating-system security updates;
- SSH key authentication and restricted sudo;
- firewall default deny;
- intrusion and authentication logs;
- container/image update process;
- uptime and certificate monitoring;
- configuration backup;
- package repository backup and snapshot retention;
- offline backup of public configuration and documentation;
- incident response and key revocation plan;
- separate secrets management outside Git.

## What Must Not Be Claimed Yet

Until verified, do not say that:

- the website domains are live;
- the package repository serves GenixBit packages;
- an ISO is available to download;
- the Store supports one-click installation;
- AI models are preinstalled;
- the GenixBit package signing key is active;
- the operating system is production ready.
